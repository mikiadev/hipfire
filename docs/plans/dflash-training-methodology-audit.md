# DFlash Training Script Audit vs Paper

**Date:** 2026-04-19
**Paper:** Chen, Liang, Liu. *DFlash: Block Diffusion for Flash Speculative Decoding.* arXiv:2602.06036 (Feb 2026).
**Reference code:** `.dflash-reference/dflash/model.py` (inference only — z-lab hasn't released training)
**DDTree paper:** arXiv:2604.12989 (tree-verify extension; **no training content**, inference only)
**Our implementation:** `scripts/dflash_train_poc.py`

## TL;DR

Our POC is a **reasonable sketch** of DFlash training but is missing three of the paper's five key training contributions. Training as-is will produce a draft that converges to some competent baseline but will under-deliver vs the paper's claimed 5-6× speedups. Before burning MI300X hours on 3.6-A3B, we need the fixes below. Target fix scope: ~250 lines in `dflash_train_poc.py`, roughly half a day.

## What the paper actually specifies

Section 4.2 "Training" lists five modifications the authors introduce:

| # | Paper's technique | What it does |
|---|---|---|
| T1 | **KV injection** | Clean prompt+response through target → fuse hidden features → inject as K/V into every draft layer |
| T2 | **Random anchor block sampling** | Sample anchor tokens from response; each anchor = first position of a masked block. Multiple blocks per sequence, concatenated with sparse attention mask |
| T3 | **Efficient long-context training** | Fix # masked blocks per sequence; randomly sample anchor positions each epoch for data augmentation |
| T4 | **Loss weighting** `w_k = exp(−(k−1)/γ)` | Earlier positions within a block weighted more — errors at position 1 invalidate all subsequent tokens |
| T5 | **Shared embedding + LM head** (frozen) | Draft reuses target's token embedding and lm_head; only draft transformer layers trained |

Plus a data specification: **target-generated responses**, not original corpus responses. Paper used ~800K samples with responses regenerated by the target model itself.

## Audit of our POC

| Technique | POC status | Effect if skipped |
|---|---|---|
| T1 KV injection | ✅ **correct** — passes `target_hidden` to `draft.forward`, the reference `Qwen3DFlashAttention` does `k_proj(target_hidden)` per layer | — |
| T2 Multi-block anchor sampling | ❌ **single-block per example** — picks ONE random start, masks one block | Dramatically fewer supervised positions per forward pass; convergence 5-10× slower; no inter-block attention-mask training (draft won't generalize to multi-block inference patterns) |
| T3 Long-context training / Flex Attention | ❌ **per-example target forward** — separate target pass per batch element, no concatenation | ~B× wasted target compute; seq_len=1024 cap means draft never sees the 6k+ agentic prompts it'll deploy on |
| T4 Loss weighting | ❌ **uniform cross-entropy** over all B−1 positions | Slower convergence; final τ likely 10-20% lower |
| T5 Shared/frozen target embedding + lm_head | ✅ **correct** — draft has no embedding/lm_head of its own; caller uses `target.model.embed_tokens` and `target.lm_head` | — |
| Arch defaults: 5 layers, 5 hidden features, block=16 | ✅ match paper | — |
| Mask token id = 248070 | ✅ match paper (reference uses it) | — |
| Data: target-generated responses | ❌ **uses corpus responses verbatim** (from kimi/glm for hermes corpus) | Distribution mismatch with target at inference time; draft learns patterns its target won't actually produce |

### Severity ranking

**Show-stopper for agentic training:**
- **T2** (single-block) is the biggest gap. Paper's "random anchor" design is what lets the draft see diverse context/masking patterns per step. Our single-block per-example wastes most of the target forward's compute and under-trains the model.
- **Data: target-generated responses**. For agentic with a distinct target (Qwen3.6-A3B), training on kimi/glm responses means the draft learns the WRONG model's output distribution. Lambda hermes has SFT traces from kimi/glm — fine for the sidecar (sidecars are response-agnostic; they care about attention patterns on the INPUT), but suboptimal for drafts (drafts learn to predict tokens).

**Meaningful but survivable:**
- **T3** (flex attention) is an optimization; skipping means training is slower, not wrong.
- **T4** (loss weighting) costs ~10-20% final τ; worth fixing but wouldn't invalidate a paper-claim result.

## Concrete fix list

To make the script paper-faithful and production-ready, in priority order:

### 1. Multi-block concat with sparse attention (addresses T2, T3, partially T4)

Replace the current `sample_batch + single-block mask` loop with:

```python
# Per sequence:
#  - Pick seq_len from corpus (tokenized, bos-aware)  [existing]
#  - Sample k anchor positions in [B..seq_len-B)
#  - For each anchor: mask positions [anchor+1 .. anchor+B)
#  - Concatenate all blocks with original prefix tokens preserved
#  - Build attention mask: within-block bidirectional + to all context features,
#    cross-block disallowed (Figure 4 of paper)
#  - One target forward (clean sequence) extracts context features
#  - One draft forward over the masked sequence predicts all blocks at once
```

Reference `.dflash-reference/dflash/model.py:279-303` shows the inference forward; we need the same attention topology for training but with multiple anchor points per sequence.

Efficiency: torch 2.5+ `flex_attention` with a block-mask callable is paper-aligned. Fallback: dense mask with a 1-hot tensor (works, 2-3× slower but still faster than the current per-example approach).

### 2. Loss weighting (T4)

```python
# w_k = exp(-(k-1)/gamma)  where k = 1..B-1 is position in block.
# gamma ~ 3-5 typically. Normalize so weights sum to 1 across positions.
gamma = 3.0
pos_weights = torch.exp(-torch.arange(B-1, device=device) / gamma)
pos_weights = pos_weights / pos_weights.sum()

# Cross-entropy per position, then weighted sum
ce = F.cross_entropy(logits.reshape(-1, V), labels.reshape(-1), reduction='none')
ce = ce.view(batch, B-1)
loss = (ce * pos_weights.unsqueeze(0)).sum(dim=1).mean()
```

`--loss-gamma` flag to tune (paper doesn't give an exact number — "γ controls the decay rate").

### 3. Target-generated responses for training data (data pipeline, not script)

This is a separate preprocessing step. Two options:

**Option A: Generate on prompts from Lambda hermes** (1-2 hours on 8× MI300X):
```
for each prompt in lambda/hermes-agent-reasoning-traces:
    regenerate response with target (Qwen3.6-A3B, temp=0)
    save {prompt, target_generated_response} to training corpus
```
This gives "on-policy" training data matched to the target model. Full 14.7k prompts at ~1k-token response averaging = ~15M tokens generated.

**Option B: Skip on-policy generation, accept the mismatch** (train faster, converge to lower τ):
Use the Lambda traces verbatim. Faster iteration but the draft learns patterns slightly misaligned with what Qwen3.6-A3B actually produces.

Recommendation: **Option B for the first validation run** (proves the pipeline, cheap). Then **Option A for the production run** once we're confident the training script is correct.

### 4. Flex Attention (T3 optimization)

```python
from torch.nn.attention.flex_attention import flex_attention, create_block_mask

def block_mask_fn(b, h, q_idx, kv_idx):
    # Within-block bidirectional + to context features, cross-block blocked
    ...

block_mask = create_block_mask(block_mask_fn, ...)
attn = flex_attention(q, k, v, block_mask=block_mask)
```

Needs torch >= 2.5. The `amd_quickdeploy.sh` torch wheel pinning should handle this (pytorch 2.11 is our target). Verify when we pull the wheel.

### 5. Other nice-to-haves

- **Gradient checkpointing on target** for big models (35B target forwards eat VRAM): `target.gradient_checkpointing_enable()`. Currently off; fine for 4B, necessary for 35B.
- **bf16 autocast** explicitly around target forward (currently implicit via `dtype=bfloat16` at load).
- **Warmup + cosine LR** already present ✓.
- **AdamW(0.9, 0.95), wd=0.01** matches paper's LLM-style hyperparameters ✓.
- **Val loss tracking** — POC has none. Add a held-out split from the same corpus to track eval loss per ckpt.

## Proposed first run after fixes

Once T2/T3/T4 are in place:

```bash
# Small-target validation (Qwen3.5-4B — trains in ~30 min on 1× MI300X)
python3 scripts/dflash_train_poc.py \
    --target-repo Qwen/Qwen3.5-4B \
    --corpus /root/calibration_corpus.txt \
    --recipe agentic \
    --block-size 16 \
    --draft-layers 5 \
    --seq-len 4096 \
    --batch-size 1 \
    --masked-blocks-per-seq 4 \
    --loss-gamma 3.0 \
    --steps 10000 \
    --out /root/dflash_4b_agentic
```

If loss drops 12 → 2-3 cleanly AND the resulting draft achieves τ > wikitext-trained-Qwen3.5-4B-DFlash baseline on a held-out hermes test split: pipeline is validated. Then scale to 3.6-A3B.
